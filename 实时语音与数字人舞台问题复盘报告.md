# 实时语音与数字人舞台问题复盘报告

## 1. 报告目的

这份报告不是“结项报告”，而是一次失败复盘和状态校正。

它要回答四件事：

1. 本次到底为什么反复改、反复错。
2. 哪些问题已经确认修复。
3. 哪些改动只是尝试过，但目前还不能算修好。
4. 后续应该怎么继续收敛，而不是继续靠感觉调参。

## 2. 当前真实状态

截至 2026 年 3 月 18 日，当前状态必须明确区分为三类。

### 2.1 已确认修复

- AI 字幕流式累积显示问题已经修复。
- 根因已经被日志和链路行为证实。
- 当前后端会按 `reply_id` 累积 `550` 文本分片，前端也会按累计文本显示，而不是只显示最后一个词或一个标点。

### 2.2 已做了较大改动，但还没有完全闭环验证

- 前端 TTS 播放链路已经从“旧 Worklet 队列模型”改成“按时间轴调度 `AudioBufferSourceNode` 顺序播放”。
- 前端已经增加“收到 `tts_end` 不立刻切回 `listening`，而是等待本地队列真正播空”的处理。
- 这些改动是针对真实根因去做的，方向是对的，但在真实页面上的最终听感还需要继续核验。

换句话说：

- **音频问题目前不能写成“已彻底修复”，只能写成“已经完成关键重构，正在收敛验证”。**

### 2.3 目前仍未修复

- 熊猫仍然没有稳定地正面朝前。
- 熊猫主体占比仍然不够，构图仍不对。
- 背景和舞台视觉没有达到可上线状态。
- 环境资源虽然已经引入了候选 HDRI，但这不等于舞台背景问题已经解决。

这一点必须写死：

- **舞台调整没有完成。**
- **环境资源接入没有形成已验收的最终方案。**

## 3. 用户实际反馈过的问题

### 3.1 视觉问题

- 熊猫侧着或斜着，不是正面朝向用户。
- 熊猫太小，主体不成立。
- 背景太抢，数字人反而变弱。
- 舞台不好看，整体视觉不成立。

### 3.2 语音问题

- 早期字幕加载不出来，TTS 正常播放。
- 后来字幕能显示，但音频听起来像“快进”。
- 主观感受包括：
  - 一句话像一秒钟就读完。
  - 前几个字像没读。
  - 标点像被单独切开。
  - 一句话前后像挤到一起。

## 4. 这次失败的核心原因

这次不是一个 bug，而是多个问题同时存在，又在没有把证据链理清之前就开始大改，最终互相污染判断。

### 4.1 先动视觉参数，后找真实坐标系

最早对熊猫问题的处理方式有明显流程错误：

- 先看截图调相机。
- 再看效果调缩放。
- 再不对就继续调背景和地台。

但正确顺序应该是：

1. 先确认模型自己的前向轴。
2. 再确认根节点 `yaw` 应该是多少。
3. 再做相机拟合和占屏比例。

如果第 1 步没做，对后面所有镜头和构图调参都会造成误导。

### 4.2 把 `tts_end` 错当成“浏览器已经播完”

后端实时链路里的 `359 / tts_end` 只说明一件事：

- 上游不再继续下发新的 TTS 音频了。

它不说明：

- 浏览器本地已经把现有音频播放完了。

如果收到 `tts_end` 就立刻把前端状态切回 `listening`，用户就会感知到：

- 状态已经结束，但音频还在播。
- 新一轮 UI 开始了，上一轮声音还没结束。
- 听感像抢句、挤句或快进。

### 4.3 旧播放器模型本身就不稳

旧的前端 TTS 路径过度依赖 [player-worklet.js](/D:/DeLu/question/frontend/public/audio/player-worklet.js) 的内部队列状态机。

问题包括：

- 队列过大时曾经会裁掉队首样本。
- 启动阈值、队列深度、播放完成判断都在 Worklet 内部耦合。
- 对很多短 chunk 和状态切换比较敏感。

这类实现一旦和 UI 状态机发生错位，就会出现：

- 前半句被吞。
- 后半句追上来。
- 听感像语速突然变快。

### 4.4 没先用数据排除“上游语速太快”

用户后面怀疑“是不是语速太快”，这个怀疑本身没有问题，但最早应该先用协议和日志算时长，而不是直接靠听感猜。

当前协议明确是：

- `pcm_s16le`
- `24000Hz`
- 单声道

时长计算公式是：

```text
duration_seconds = bytes / (24000 * 2)
```

用户贴的真实日志里，一轮回复包含多块 `352` 音频 chunk，总字节数远大于 1 秒语音应有的大小。

所以结论是：

- 这不是上游真的 1 秒把整句读完。
- 这是前端播放时序和状态处理让音频“听起来像 1 秒读完”。

### 4.5 背景设计同时在 Three.js 和 CSS 两边抢主体

背景失败不是元素不够，而是层次错了。

场景里同时存在：

- Three.js 里的环、光幕、地台、背板。
- CSS 里的整页渐变、发光遮罩、底光。

结果是：

- 背景不是衬托层，而是和数字人争主视觉。
- 3D 层和 CSS 层一起抢镜，数字人反而最弱。

## 5. 本次踩过的坑

### 5.1 只凭截图调舞台和镜头

截图只能说明“看起来不对”，不能替代模型坐标、包围盒和前向轴验证。

### 5.2 把字幕问题和音频问题混在一起

二者都属于实时会话，但不是同一根因：

- 字幕问题来自 `550` 文本流分片被覆盖。
- 音频问题来自前端播放链路和状态机错位。

### 5.3 过早优化延迟，牺牲了正确性

旧播放器默认优先级更像：

- 队列不要太长
- 延迟越低越好

这在当前项目里是错的。当前项目正确优先级应该是：

1. 保证整句按顺序完整播放。
2. 再谈低延迟。

### 5.4 在主体没有锁住前先做大背景

熊猫还没摆正、还没放大，就先做大舞台，结果只会把错误放大。

正确顺序应该是：

1. 主体朝向
2. 主体占屏
3. 镜头安全区
4. 背景衬托

### 5.5 文档写得比真实状态更超前

这是这次必须明确承认的问题。

之前的复盘文字里，把一些“已经开始实施的方向”写成了“最终修复方案”，把一些“理论上应该达到的效果”写得像“当前已实现效果”。

这会直接误导判断，尤其在舞台和背景这类还没验收的部分上。

## 6. 已确认的根因与对应处理

### 6.1 字幕链路

根因：

- 上游 `550` 文本是流式分片。
- 如果前端或后端按片段覆盖，就只会剩最后一段。

已完成处理：

- [session_manager.py](/D:/DeLu/question/backend/app/session_manager.py) 按 `reply_id` 维护累计文本。
- [upstream.py](/D:/DeLu/question/backend/app/realtime/upstream.py) 的 mock 行为补成更接近真实上游的分片流。
- [test_realtime.py](/D:/DeLu/question/backend/tests/test_realtime.py) 增加了对应测试。
- [useRealtimeSession.ts](/D:/DeLu/question/frontend/src/hooks/useRealtimeSession.ts) 和 [VisitorPage.tsx](/D:/DeLu/question/frontend/src/pages/VisitorPage.tsx) 也按累计文本显示字幕。

当前判断：

- **字幕问题可以视为已修复。**

### 6.2 音频链路

已确认根因：

- 旧 Worklet 队列策略曾经裁掉前段音频。
- `tts_end` 和浏览器真实播放完成被错误地当成同一个时刻。

已完成处理：

- [audio.ts](/D:/DeLu/question/frontend/src/lib/audio.ts) 已改成基于 `AudioBufferSourceNode` 的时间轴顺序调度。
- [useRealtimeSession.ts](/D:/DeLu/question/frontend/src/hooks/useRealtimeSession.ts) 已增加本地排队时长和 `tts_end` 协同控制，不再在服务端一发结束信号就立即回到 `listening`。

当前判断：

- **音频根因分析已经比较清楚，修复方向也已经落地。**
- **但“最终听感已完全恢复正常”这件事，还需要继续以真实页面播放效果为准。**

## 8. 为什么之前那版文档不对

问题主要有三类。

### 8.1 把“修复方向”写成了“修复结果”

例如：

- 写成“熊猫朝向与构图修复”
- 写成“背景修复”

但实际上这些部分仍然没通过用户验收。

### 8.2 把“应该达到的状态”写得像“当前状态”

例如：

- “熊猫正面朝前”
- “背景不再比数字人更抢眼”

这些内容在设计目标上没有问题，但不能写成当前事实。

### 8.3 没把“环境资源接入”和“舞台完成”分开

引入 HDRI 是手段，不是结果。

如果最终舞台还是侧身、主体还是小、背景还是丑，那么文档就不能借“已接入资源”来暗示方案已经成立。

截至当前，比较准确的项目状态应该写成下面这样。

### 9.1 可以确认的

### 9.2 不能确认完成的

## 10. 后续继续修时的正确顺序

### 10.1 音频

先补运行时诊断日志，至少记录：

- chunk 字节数
- chunk 时长
- 调度开始时间
- 调度结束时间
- 当前本地累计排队时长

这样就能明确判断：

- 是上游发短了
- 还是前端调度错了

### 10.2 熊猫

单独做模型调试视图，明确显示：

- 包围盒
- 根节点
- XYZ 轴
- 当前 `yaw`

在没有确认模型真实前向轴之前，不再继续凭截图调相机。

### 10.3 背景

背景必须回到最小方案，只允许：

- 深色远景
- 一个弱主背板
- 一个地台
- 一组服务于主体的灯光

在主体没锁定前，不再继续加大环、大罩子、大面积 CSS 光层。

## 11. 总结

这次失败，不是因为“调参还不够细”，而是因为：

- 根因分析和视觉调参顺序颠倒了。
- 播放完成和状态完成被混为一谈。
- 舞台和背景在主体没有锁住前就被过度展开。
- 文档表述一度超前于真实修复状态。

当前最准确的结论应该是：

- **字幕问题已修复。**
- **音频问题已经完成关键级重构，但还要继续用真实页面收敛验证。**
- **熊猫朝向、主体构图、背景舞台，目前仍未修好，不能写成已完成。**
- **环境资源目前只能算候选资产接入，不能算舞台方案落地。**
