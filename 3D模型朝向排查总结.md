# 3D模型朝向排查总结

## 1. 背景与问题现象

本次问题的核心现象是：

- 页面中的 `panda-v2.glb` 长时间以侧身或半侧身状态面向用户。
- 前面已经做过多轮修改，但视觉上仍然像“没生效”。
- 用户的直接目标非常明确：模型必须正对用户，不能再侧着。

这类问题表面上看像是“模型 rotation 不对”，但实际排查后发现，它通常不是单点问题，而是由以下几个环节叠加造成：

1. 当前到底加载的是哪个模型。
2. 当前模型到底命中了哪个 preset。
3. preset 里的 `modelYaw`、`cameraPosition`、`cameraTarget` 谁在起主导作用。
4. 运行时有没有每帧把 rotation 重写回去。
5. 本地启动链路有没有继续使用旧模型或旧构建。

本次就是一个典型案例：最开始看起来像是“模型朝向锁死”，实际最后的决定因素更多是“相机一直站在模型的侧前方”，所以不管怎么改一点点 rotation，最终视觉上都很像还是侧着。

---

## 2. 最终结论

### 2.1 真正控制朝向的主链路

当前项目里，模型朝向不是只在一个地方控制，而是有一整条链路：

1. `frontend/src/avatar/presets.ts`
   - 负责决定当前模型使用哪套 `AvatarPreset`。
   - 这里定义了：
     - `modelYaw`
     - `rigRootRotation`
     - `cameraPosition`
     - `cameraTarget`
     - `cameraFov`
     - `heightFill`
     - 其他舞台 framing 参数

2. `frontend/src/components/AvatarStage.tsx`
   - 在模型加载后，先用 `preset.modelYaw` 参与包围盒拟合和居中。
   - 然后在 `useFrame()` 中每一帧都根据 preset 和动画偏移重写模型根节点 rotation。

3. `frontend/src/avatar/adapter.ts`
   - 负责“活起来”的骨骼动画和少量位移/转动偏移。
   - 这部分如果给了 `rotation[1]`，就会在运行时持续叠加到模型 Y 轴朝向上。

### 2.2 本次问题的本质

最终确认，本次“模型总是侧着”的根因不是：

- 不是 `adapter.ts` 在偷偷把它转回去。
- 不是 `CameraRig` 在控制模型本体 rotation。
- 也不再是旧的 `default-avatar.glb` 在干扰当前结果。

真正的原因是：

- `panda-v2.glb` 这个资源的“视觉正前方”并不沿着常见的默认相机正前方向。
- 也就是说，这个模型的“脸朝向”更接近场景的 `+X` 方向。
- 如果相机站在 `+Z` 方向或者 `+X/+Z` 的斜前方，用户就会看到侧身或 3/4 视角。

因此，这个问题最后的稳定修复方式不是继续硬拧 `modelYaw`，而是：

- 保持 `modelYaw = 0`
- 把 `cameraPosition` 和 `cameraTarget` 调到模型真正的“正面轴”上

也就是把相机移动到模型的正前方，而不是继续在模型本体上来回试角度。

## 3. 关键证据与代码定位

## 3.1 preset 路由位置

文件：`frontend/src/avatar/presets.ts`

关键位置：

- 第 33-49 行：`pandaPreset`
- 第 51-56 行：`resolveAvatarPreset()`

当前逻辑：

```ts
if (normalized.includes("default-avatar.glb") || normalized.includes("熊猫") || normalized.includes("panda")) {
  return pandaPreset;
}
```

这意味着：

- 只要模型 URL 包含 `panda`，最终就一定会走 `pandaPreset`。
- 所以前面如果改的是 `defaultPreset`，或者改的是别的 preset，页面上很可能根本不会生效。

---

## 3.2 模型被每帧重写 rotation 的位置

文件：`frontend/src/components/AvatarStage.tsx`

关键位置：

- 第 143-146 行：拟合包围盒时使用 `preset.modelYaw`
- 第 222-225 行：运行时每帧写回模型 rotation
- 第 231 行：group 初始 rotation 也使用 `preset.modelYaw`

关键代码：

```ts
fitRoot.rotation.y = preset.modelYaw;
```

```ts
root.current.rotation.set(
  rigOffset.rotation[0],
  preset.modelYaw + rigOffset.rotation[1],
  rigOffset.rotation[2],
);
```

这说明：

- 就算只改了 JSX 里的初始 `rotation`，也没用。
- 因为到了 `useFrame()`，模型会再次被写回 `preset.modelYaw + rigOffset.rotation[1]`。
- 所以模型朝向要稳定生效，必须改真正的源头：preset 或动画偏移，而不是临时节点 rotation。

---

## 3.3 动画系统并没有在强制 Yaw 回正/回侧

文件：`frontend/src/avatar/adapter.ts`

关键位置：

- 第 320-330 行：`driveAvatar()` 返回最终 runtime 偏移

当前 Y 轴返回值：

```ts
rotation: [
  0.008 * breathe + ...,
  0,
  0.008 * idleSway,
],
```

这里第二个值就是 `rotation[1]`，当前明确是 `0`。

这说明：

- 当前动画逻辑不会主动在 Y 轴上把模型扭到侧面。
- 所以“我明明改了角度但它又自己转回去”并不是 `adapter.ts` 导致的。
- 真正主导的是 preset 和 camera framing。

---

## 3.4 CameraRig 只控制相机，不控制模型

文件：`frontend/src/components/AvatarStage.tsx`

关键位置：

- 第 426-439 行：`CameraRig`

核心逻辑：

```ts
camera.lookAt(targetVector);
```

这段代码的意义是：

- 相机始终看向某个目标点。
- 它不会改模型自身 rotation。

所以：

- 如果视觉上像是模型侧着，有时其实不是模型转错了，而是相机站错了位置。
- 本次问题最终就是这个方向。

---

## 3.5 启动链路已经固定到 `panda-v2.glb`

文件：`start-local.ps1`

关键位置：

- 第 6-8 行：模型路径
- 第 50-55 行：模型复制逻辑
- 第 64 行：`DEFAULT_AVATAR_URL = "/models/panda-v2.glb"`
- 第 113-116 行：启动前自动执行 `npm run build`

这部分后续非常重要，因为它解释了为什么一开始有时看起来像“改了没生效”：

- 如果启动脚本还在复制旧模型，或者默认 URL 还指向旧资源，就算前端代码改对了，也可能看不到正确结果。
- 现在这条链路已经统一到 `panda-v2.glb`，不会再因为默认模型混乱导致判断失真。

## 4. 为什么前面的多次修改看起来没用

这次问题难排，主要是因为前面几种“常见修法”都容易造成假象。

### 4.1 改错 preset

如果改的是 `defaultPreset`，但当前实际走的是 `pandaPreset`，页面不会有任何变化。

### 4.2 只改初始 rotation，没有改每帧回写源头

`AvatarStage.tsx` 在 `useFrame()` 里每帧都写：

```ts
preset.modelYaw + rigOffset.rotation[1]
```

所以只改初始 JSX `rotation`，最终会被覆盖。

### 4.3 把“相机视角问题”误判成“模型朝向问题”

这一点是本次最关键的误区。

前面最像真相的一种错误判断是：

- “模型本体一定被强制转回侧身了”

但真正排到最后发现：

- 模型本身不一定被转错。
- 很多时候是相机始终站在模型的斜侧前方。
- 于是用户看到的结果自然像“模型又侧过去了”。

### 4.4 旧模型 / 旧进程 / 旧构建也会制造假象

如果启动链路里还残留旧模型，或者页面没有刷新到最新 bundle，也会出现：

- 代码确实改了
- 但屏幕上看起来完全没变

这也是为什么这次必须把：

- 模型资源路径
- preset 命中逻辑
- build 产物
- 本地页面截图

一起串起来验证，而不能只看某一处代码。

## 5. 本次最终修正内容

本次最终生效的修正落在：

文件：`frontend/src/avatar/presets.ts`

最终的 `pandaPreset` 为：

```ts
const pandaPreset: AvatarPreset = {
  modelYaw: 0,
  rigRootRotation: [0, 0, 0],
  positionOffset: [0, 0.02, 0],
  scaleMultiplier: 1.16,
  cameraPosition: [3.12, 0.9, 0.04],
  cameraTarget: [0.08, 0.62, 0],
  cameraFov: 27,
  heightFill: 0.7,
  floorY: -0.16,
  floatIntensity: 0.01,
  floatSpeed: 0.24,
  shadowScale: 3,
  shadowOpacity: 0.48,
};
```

### 5.1 这组参数的意义

#### `modelYaw: 0`

- 不再继续通过绕 Y 轴硬转模型去“碰运气”。
- 保持模型自身基础朝向稳定。

#### `cameraPosition: [3.12, 0.9, 0.04]`

- 把相机放到接近模型 `+X` 方向的正前方。
- 同时把 `z` 几乎压到 `0`，避免继续形成斜 3/4 角度。

#### `cameraTarget: [0.08, 0.62, 0]`

- 让相机看向胸口和头部之间的正面区域。
- 避免因为 target 偏移过大，导致镜头又从侧前方看过去。

### 5.2 为什么这次终于生效

因为这次不再把问题当成“模型 rotation 死锁”，而是按真实问题处理：

- 模型前脸轴向与相机默认视角不一致
- 那就把相机放到它真正的正前方

这比继续试 `Math.PI / 2`、`Math.PI`、局部骨骼旋转、运行时 group rotation 更稳定，也更不容易被 `useFrame()` 逻辑反向覆盖。

## 7. 这次排查得到的经验结论

以后项目里再遇到“模型怎么改都侧着”时，建议按下面顺序排：

### 第一步：先确认当前到底加载的是哪个模型

不要默认以为页面上跑的是你刚改的资源。

先确认：

- 当前 `avatarUrl` 是什么
- 当前命中了哪个 preset
- 启动脚本有没有复制旧文件

### 第二步：确认 rotation 有没有被每帧重写

重点看：

- `useFrame()`
- 动画适配层返回的 rotation offset
- preset 是否在 runtime 中被持续使用

### 第三步：区分“模型方向错”还是“相机站位错”

这是最关键的一步。

判断方法：

- 如果 `rotation[1]` 没有被持续写入，但画面还是侧着，优先怀疑相机站位。
- 如果相机已经在合理正前方，而模型仍明显侧身，再去改 `modelYaw` 或 `rigRootRotation`。

### 第四步：只保留一个稳定控制点

对于这种舞台型展示页，最好只让一个主控制点决定“面朝哪里”：

- 要么由 `modelYaw` 决定
- 要么由 `cameraPosition/cameraTarget` 决定

不要多个地方同时拧，不然很容易出现：

- 某处改了有效
- 另一处又把它抵消
- 最终大家都觉得“怎么改都没用”

---

## 8. 以后如果还想继续微调，应该改哪里

### 想让人物继续左右转一点

优先改：

- `frontend/src/avatar/presets.ts`
- `cameraPosition`
- `cameraTarget`

不要先去改：

- `AvatarStage` 的临时 group rotation
- `adapter.ts` 的 runtime yaw

### 想让模型本体真正转身

才去改：

- `modelYaw`
- 或 `rigRootRotation`

适用场景：

- 更换了新 GLB
- 新模型前脸轴向和当前熊猫不同

### 想让人物更“活”但不破坏正面朝向

继续保持：

- `adapter.ts` 中 `rotation[1] = 0`

这样可以让它有点呼吸、点头、说话、挥手，但不会在 Y 轴上慢慢拧成侧身。

## 10. 一句话版本结论

这次问题最后不是“模型一直被强制转侧身”，而是“相机一直站在模型的侧前方”。真正稳定的修法是把 `panda-v2.glb` 的镜头移到它真实的正前方轴上，而不是继续在运行时反复硬拧模型 Y 轴 rotation。
